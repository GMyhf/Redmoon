## 0. 为什么先做这个

我们每次击杀掉 `level×2` 金币，**没有任何长期消耗池**——经济必然通胀。红月的核心留存循环不是打怪，是**赌**：
把装备往上精炼，失败要付代价，保护符是失败保险。私服把保护符标价 $2 单卖，是那个仓库里最硬的需求信号。
这同时也是「意志（will）」的出口（人已拍板：`will` 留作精炼货币）。

## 2. 反推成功率曲线（本规格最有价值的一步）

礼包捆绑的保护符数量是**成功率的指纹**：商家最清楚玩家平均要烧掉几张。

假设一条自然的递减曲线 **90% / 70% / 50% / 30%**（0→1 / 1→2 / 2→3 / 3→4），
则爬到各阶的**期望尝试次数** = Σ(1/p)：

| 目标阶 | 期望尝试次数 | 向上取整 | 礼包实际捆绑 | 对应礼包 |
| --- | --- | --- | --- | --- |
| → Stage 2 | 1/0.9 + 1/0.7 = **2.54** | 3 | **3** | 入门 $50 |
| → Stage 3 | + 1/0.5 = **4.54** | 5 | **5** | 中级 $100 |
| → Stage 4 | + 1/0.3 = **7.87** | 8 | **8** | 高级 $150 |

**三个礼包的捆绑量全部命中。** 一条由整十数构成的自然曲线，独立地预测出了 3/5/8 三个数字。

**诚实的反证**：3、5、8 恰好是连续的斐波那契数，也可能只是「递增的好看数字」，与成功率无关。
我无法排除这个解释。所以正确的表述是：**90/70/50/30 与观察到的捆绑量高度自洽，可以作为我们的起点，
但它不是从红月提取出的事实。** 我们的曲线终究要按我们自己的经济来调——见 §5 未决项。

## 3. 我们的设计（长在现有代码上）

### 3.1 物品结构

现有物品在 `loot.js` 里数值是**掉落时摇好存下来的**（`bonuses[stat] = 30 + itemLevel*3` 等），不是实时算的。
因此精炼**绝不能重摇 bonuses**，否则玩家会靠精炼刷词条。方案：存阶段，读取时确定性放大。

```js
// 新增字段（默认 0，老存档读出来就是 0，无需迁移）
item.refine = 0..4

// 所有数值读取走一个 helper，确定性、无随机：
//   effective = stored * (1 + REFINE_STEP * item.refine)
// REFINE_STEP = 0.15  →  满阶 +60%
```

对齐红月的 `[Original Level]` / `Current Upgrade Level` 双显：我们的 `item.level` **保持不变**（它就是 Original
Level），UI 上把精炼阶数单独显示（`+1`…`+4`）。不新增 `originalLevel` 字段——我们的 level 本来就不动。

**适用范围**：`tier >= 3`（rare 及以上）。理由：sunset 是 tier 7、0.9% 掉率、全服同时只允许 1 个在地上——
只让 sunset 可精炼等于把整个系统锁死成终局内容，几乎没人碰得到。rare+ 才能让精炼成为**贯穿全程**的消耗池。

### 3.2 成功率与代价

| 阶段 | 成功率（起点） | 失败惩罚 | 护炉印可挡 |
| --- | --- | --- | --- |
| 0→1 | 90% | 无（仅损失货币） | — |
| 1→2 | 70% | 掉 1 阶 | ✅ |
| 2→3 | 50% | 掉 1 阶 | ✅ |
| 3→4 | 30% | 掉 1 阶 | ✅ |

- **失败惩罚 = 掉 1 阶**。红月的惩罚是「物品丢失累积经验」，但我们的装备没有经验，直接照搬无意义。
  掉阶是同一类惩罚（抹掉已投入的进度）在我们系统里的等价物，且正好给护炉印一个存在理由——
  **没有惩罚就没有保险，没有保险就没有赌，整个循环就塌了。**
- **不做「物品碎裂」**。韩式 MMO 常见，但对我们过于严苛：我们没有交易、没有邮件、没有仓库（都在 P4），
  玩家丢了装备没有任何补救渠道。等 P4 经济落地后可以再议。

  > **2026-07-18 销账**：**P4 已经落地**——银行（T-033）、邮件（T-034）、寄卖行（T-035）全部交付，
  > 所以上面那句「再议」的触发条件**已经满足**，但从没人回来议。**此处正式结论：仍然不做物品碎裂。**
  > 理由已经变了，记明白：不再是「玩家没有补救渠道」（现在有了——寄卖行可以买回同类装备），
  > 而是**回廊已经在 T-039 承担了「会丢东西」这个角色**（击杀掉落未精炼的随身物品）。
  > 再给精炼加一个碎裂，就是同一种焦虑收两次费，且精炼的赌性已由「掉阶 + 护炉印」表达充分。
  > **这是设计判断，若人不同意可推翻**——但不要再让它以「等 P4」的形式悬着。

### 3.3 货币与消耗

- **意志（will）** —— 主消耗。已在击杀时 `player.will += mob.level` 累加并持久化，**这是它的出口**。
  成本随物品等级和目标阶数上升：`cost = itemLevel × (stage+1) × WILL_PER_LEVEL`。
- **金币** —— 并行消耗，把我们缺的长期金币池补上。
- **护炉印** —— 失败保险，消耗品。失败时自动消耗一张、阶数不掉；玩家可选择不带。

**护炉印从哪来？** 红月卖真钱（$2），我们没有支付。三个候选，我倾向 (a)：
- **(a) 黑市商人·影三 收复苏露**（推荐）：他已经存在、已经收复苏露（遗物盒 3 露）。复苏露是我们的稀缺货币
  （2% 掉率），拿稀缺货币换失败保险，符合它「保险」的定位，也给复苏露第二个出口。
- (b) 纯金币：简单，但金币不稀缺，护炉印会变成无脑常驻，赌的张力消失。
- (c) 怪物掉落：又多一个掉落池，且和 uniq/sunset 的 maxActive 机制冲突。

### 3.4 交互位置

**锻匠·坤铁**（已存在，`SHOPS.smith`，已卖「定制装备」120 金）。精炼是他的第二项服务，天然契合。
`SHOP_RANGE = 130` 的距离校验已是服务端权威，直接复用。

### 3.5 服务端权威（红线）

**摇骰子必须在服务端。** 新增命令 `refine`：

```
客户端 → 服务端：{ type: "refine", itemId, useProtection: bool }
服务端：校验（玩家活着 / 在坤铁 SHOP_RANGE 内 / 物品在背包 / tier>=3 /
              refine<4 / will 足够 / gold 足够 / 带护炉印则库存足够）
       → 扣货币 → world.rng() 判定 → 改 item.refine → _emit("itemRefined", {...})
失败路径：WorldError(code, message) → 网关转成 error 消息
```

客户端**只发意图**，成功率、判定、结果全在 tick 边界由服务端决定。新增错误码：
`REFINE_MAX_STAGE` / `REFINE_TIER_TOO_LOW` / `NOT_ENOUGH_WILL` / `NO_PROTECTION` / `SHOP_TOO_FAR`（已有）。

### 3.6 协议影响

- `refine` 命令：**新增消息类型**，不破坏既有字段。
- `item.refine` 字段进快照：`codec.js` 的物品编码要加一个 u8。**这是破坏性改动 → `PROTOCOL_VERSION` 2 → 3**，
  并同步 `docs/ARCHITECTURE.md` 协议表 + 两个客户端（浏览器 + Godot）。
- 顺带：若同期要删/改其他字段，应合并进这一次 v3，避免连续两次升版本。
