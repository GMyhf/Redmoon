# 性能实测与决策记录

2026-07-12，实测环境：本仓库开发机（Linux，Node 22 / Godot 4.3 headless）。
所有数字来自可复现脚本；结论按"出现某信号才动工"的方式记录，避免过早优化。

## 服务端（120 怪同图，玩家各带 30 格背包 + 8 好友）

| 同图玩家 | tick | 快照构建/广播 | + JSON 序列化 | + binary1 序列化 |
| --- | --- | --- | --- | --- |
| 20 | 0.45ms | 2.1ms | 20.6ms（52.6 KiB/连接） | 18.5ms（19.2 KiB/连接） |
| 50 | 0.54ms | 2.7ms | 150.9ms（67.8 KiB/连接） | 74.3ms（22.1 KiB/连接） |
| 100 | 1.0ms | 2.6ms | 241.9ms（93.3 KiB/连接） | 88.5ms（26.9 KiB/连接） |

- 世界模拟（tick）与共享构建都健康；成本集中在**每连接的序列化**。
- binary1 编码器最初逐字段分配 Buffer + concat，比 JSON 慢 2.8 倍；重写为
  预分配顺序写后提速 6.5 倍（579ms → 88.5ms @100p），现在比 JSON 快 2.7 倍
  且带宽约为 1/3.5。

**决策**

- 增量快照：**暂缓**。当前在线规模（≤20 同图）序列化约 20ms/广播（10Hz ≈ 2%
  单核）。触发条件：单图并发 > 30，或单次广播序列化 > 50ms。
- 对象池：**不做**。tick 在 100 玩家下仅 1ms，无 GC 压力证据。
- JSON 路径的共享部分预字符串化：**记录备选**，优先级低于增量快照。

## 压力测试（tools/stress.mjs，50 bot 全同城镇图，systemd 实例）

| 阶段 | tick 达成率 | 快照间隔 p50/p95/p99 | 服务器 CPU |
| --- | --- | --- | --- |
| 优化前 | 52%（世界慢动作） | 181 / 342 / 401 ms | 85% |
| 共享序列化 + tick 补偿后 | 97% | 142 / 256 / 370 ms | 84% |

- 共享序列化：同图实体数组每次广播只 stringify/打包一次，每连接仅序列化本人条目再拼接（JSON 与 binary1 双路径，等价性由测试锁定）。微基准：JSON 100 玩家 242→106ms/广播，binary1 88.5→15.6ms。
- tick 循环改为时间累积器：事件循环拥堵时按真实流逝补步（上限 5 步），世界时间不再随负载变慢。
- 50 个 bot 挤在同一张图是最坏情形（真实玩家分散十图）；当前判定：单图容量 ≈ 40 人（p50 达标），事件已按图作用域投递。触发增量快照的新条件：共享序列化后单图并发 > 60。

## Godot 客户端（headless 计时 GDScript 算法层）

| 项 | 成本 |
| --- | --- |
| 750 可视地砖着色（换图冷启动） | 2.99ms（仅首帧） |
| 同上（暖缓存，每帧） | 0.35ms |
| 150 实体画家排序（每帧） | 2.49ms |
| 20 个等距环 ×24 段（每帧） | 0.19ms |

- 算法层每帧总开销 < 3ms；真正的未知数是 RenderingServer 的 draw 调用提交
  （~800 多边形/帧），headless 测不到，需真机 FPS 数据。

**决策**

- 静态地表烘焙（把地砖画进单张 Image 纹理按图缓存）：**不急**。若真机掉帧
  且定位到 draw 提交，这是第一预案。
- 实体排序优化（脏标记/分桶）：实体 > 200 时再做；当前 ~50 实体成本 <1ms。
