# 游戏改进计划 · 对照 RedMoon Fantasy

> 制定日期：2026-07-16。参照对象：`/home/ubuntu/hongfei/RedMoon-Fantasy` —— 韩式经典 MMORPG
> **红月（RedMoon 3.944e）** 的私服发行仓库（客户端 DLL + `DATAs`/`RLEs` 素材 + PHP 门户站）。
> 本文只做**规划**，不改代码。任务真正开工前仍需按 `collab/README.md` 的约定进 `collab/PLAN.md` 认领。

## 一句话结论

CRIMSON RELAY 已经继承了红月的**骨架**（1000 级上限、转生、四维、sunset/unique 稀有度、
地球→沙漠→雪山→星舰→天空城的地带推进），但缺的是红月真正留住玩家的**三个循环**：
**PvP + 荣誉**、**装备精炼赌博（Sunset 0-4 + 保护符）**、**军团与阵营**。
我们现在有一条很扎实的「刷怪→升级→转生」单机线，没有任何**玩家之间**的结构。

## 1. 参照对象是什么

红月是 2000 年代初的 640×480 / 16 位色 DirectDraw 网游，这个私服仓库有三部分：

| 部分 | 内容 | 对我们的价值 |
| --- | --- | --- |
| `RedMoon3944/DATAs`、`RLEs` | 222 张地图、1108 条物品定义、**386 个技能**、96 种怪物/NPC、约 19.2 万帧精灵 | **系统清单**：一个成熟韩式 MMO 到底有哪些系统、内容规模是多少 |
| `User Interface Design/` | PHP 门户（开户 / 排行 / 在线 / 商店 / 转生 / 工具） | **数据库 schema 即设计文档**：字段名直接暴露了玩法（`SubQuestGiftFame`=转生数、`STotalBonus`=排行分、`SETimer`=buff 串） |
| 客户端 DLL、注册表补丁 | WinXP 兼容层、dgVoodoo 式 D3D 垫片 | 无价值，**反面教材**（见第 6 节） |

素材本身不可直接复用（`.rle`/`.rmd` 是私有格式，且是别人的美术资产）。**可复用的是设计，不是文件。**

## 3. 与现有路线图的关系（重要边界）

`README.md` 路线图 **#4 明确写着「PvP、赛季、拍卖行与更多地图暂缓」**，前置是 #1 PostgreSQL 生产化
（真实迁移、备份恢复、连接故障、审计保留、恢复/轮换演练）尚未收尾。本计划**不推翻这个决定**，而是把 #4
拆成「被闸门挡住的」和「没被挡住的」两堆：

- **被 #1 挡住**：PvP、军团、阵营、交易、邮件 —— 它们都要持久化新的账号级数据、要抗滥用、要经得起
  长期运行。在 Postgres 演练收尾前排期只会制造迁移债。
- **不被挡住**：转生数值失衡、`will` 死资源、R/C 换皮、装备精炼、消耗品、排行榜 —— 这些跑在既有的
  内存战斗世界 + 既有账号字段里，不引入新的运营风险。

**所以下面 P0/P1 现在就能做，P2 之后需要人先确认 #1 的闸门开了。**

## 4. 分阶段计划

### P0 · 立即修正（存量债，小改动）· **已完成**

1. **转生数值失衡（真 bug，不是特性）** —— `REBIRTH_LEVEL = 10`，而每次转生给
   **永久 +6 属性点 / +12% 生命 / +15% 伤害且无限叠加**（`definitions.js:14-17`）。
   10 级几分钟就能刷到，玩家可以在低级区无限转生把数值叠爆，1-1000 的等级阶梯直接失去意义。
   红月的门槛是 **1000 级**，正因为奖励是永久的。
   建议：大幅提高门槛（或按转生次数递减收益），并把 `LEVEL_CAP` 的意义找回来。**这条应先于一切新系统。**
2. **`will` 是死资源** —— 每次击杀 `player.will += mob.level`（`world.js:2408`），持久化了、也序列化给客户端了，
   但全仓没有任何消费方。要么给它一个出口（P1 的精炼货币是天然人选），要么删掉。
3. **R/C 两格换皮**（T-012 已完成）—— 8 个职业的 R 和 C 走同一份 `shared:r`/`shared:c` 行为，只有名字不同。
   **实做时发现这不是「纯粹的品质提升」，而是一个数值 bug**：`shared:r` 按 `力量+精神` 成长、`shared:c` 按
   `敏捷` 成长，但自动加点按 `ALLOC_WEIGHTS` 分配——壁垒者只有 **8.8%**、焚灵 **9.6%** 的加点进敏捷，
   **五个职业的 C 属性加成近乎失效**。16 个槽位现已各自拥有行为并按本职业主属性成长，最低占比 8.8% → 35.1%。

### P1 · 装备精炼 + 保护符 · **已完成**（T-011，规格见 `docs/P1_REFINE_SPEC.md`）

红月的核心留存循环不是打怪，是**赌**：装备升到 Sunset 0→4，失败会掉级/碎裂，**保护符**是失败保险
（私服卖 $2，说明它是最硬的需求）。这是金币和物品的**主要消耗池**——没有消耗池的经济必然通胀，
我们现在每次击杀掉 `level×2` 金币且**没有任何长期消耗**。

我们的物品系统（`bonuses` / `damageBonus` / `hpBonus` / `attackFormula`）已经够支撑，需要新增的只是：
阶段字段、成功率曲线、失败惩罚、保护符物品、以及服务端权威的判定（**绝不能让客户端摇骰子**）。
自包含、不碰账号 schema、不被 #1 闸门挡住。

### P2 · PvP + 荣誉（人已于 2026-07-17 确认 #1 闸门）

按风险从低到高分三步，**每步可独立上线**：

1. **决斗场** · **已完成**（T-016）：双方同意、隔离地图、无掉落、无荣誉 —— 只验证「伤害能路由到玩家」这条最核心的链路。
   实做时发现两个不在原计划里的硬点：① 隔离必须靠**从决斗成员列表解析对手**，而不是按距离/位置扫描玩家，
   否则另一张图上同坐标的旁观者会被打到；② 竞技场**必须有自己的边界**——四条移动路径全都钳制到
   `this.width/height`（4800×2700），不改就等于竞技场没有墙。
2. **荣誉** · **已完成**（T-017）：~~把 Eclipse 专属的 `reputation` 提升为全局属性，PvP 击杀驱动~~ —— **这个原始设想在实做时被推翻**：
   ① 决斗不给荣誉（同意的 PvP 可互喂），野外 PvP 在第三步 → 照做只会交付一个永不变动的死数字，
   正是 P0 批评 `will` 的形状；② `reputation` 是 Eclipse 主动选择的构筑轴（符号翻转整套技能），不是社会声望。
   **改为独立字段 + PvE 来源 + 门禁消费方**，依据来自红月自身的字符串表：`Killing Red monsters generates positive karma!`
   （荣誉有 PvE 来源）、`Honor is insufficient to upgrade.`（门禁精炼）、`To create army you must be at level %u and honor %u`
   （门禁军团，P3 前置）。**荣誉是门槛不是货币，只检查不扣除。** 第三步再加 PvP 的负向驱动。
3. **战斗区** · **已完成**（T-020）：开放 PvP 的**独立地图**（红月的做法是隔离，不是全图开放）。
   **两处偏离原计划，均由人拍板**：① 计划写「有掉落」，实做改为**只赌金币与荣誉、装备与经验不掉**——
   我们没有银行/邮件/交易（都在 P4），丢了的装备零回收，而红月有这些兜底；② 反刷不靠红月的 PK 惩罚计时，
   而是**荣誉转移以对方实际拥有量为上限**：小号无荣誉可夺、互喂零和、只有赢真正有声望的人才涨。
   荣誉借此闭环——来源（精英）与风险（被抢）在同一张地图上。**协议借这一步升到 v4**（荣誉转为公开）。
   **未做**：红月的 `BattleMatch`（定时双队赛事、积分、奖品、等候室、分阶段）需要阵营（P3）与事件调度器（P5），
   不属于这一步。

### P3 · 军团（Army）+ 阵营

顺序：**军团 → 阵营 → 要塞 → 攻城**，跳步会做出没人打的空竞技场。

1. **军团** · **已完成**（T-021）：~~`Name`/`Commander`/`Camp` 三个字段就是红月军团的全部~~ ——
   **实做时发现远不止**：红月的字符串表还有 `Commander or Lieutenant`（军衔）、
   `wants to entrust his position to you. Do you accept the offer?`（转让需接掌）、
   `That Army name already exists.`（团名唯一）、`Choose the Army Hall floor you wish to rent.` +
   `The army hall rent hall is due %d-%d-%d. You must pay %u.`（**要塞是租的，有周期租金**——一个金币消耗池，
   不是「打下来就归你」）、以及结盟与宣战。已做：建立（等级+荣誉门禁，荣誉的第二个读者）、军衔、招募需同意、
   逐人、转让需接掌、解散、军团频道、公开的团名与军衔。
   **实现上零 schema 迁移**：军团不是一张表，而是「所有声明该名字的账号」——账号记录本就是自由 JSON。
2. **阵营** · **已完成**（T-024）：自由邦 / 契约同盟，**属于军团而非个人**（红月 `tblArmyList1.Camp` 在军团表上，
   地图里两座镜像要塞群各属一方）。**没有做成「等攻城才生效的标识」**——那会是第三个死字段（前两次是 `will` 与荣誉）。
   它当天就有用：**血斗回廊里同阵营互不可攻击、跨阵营才能开火**，回廊从混战变成两军交战；无阵营者仍人人可打。
   **一经宣誓不可更改**——可切换的阵营是逃生按钮；改换门庭属于攻城那一层。**协议借此升到 v5**（阵营转公开）。
3. **要塞**：红月的 Army Hall 是**按层租用**、周期付租的。
4. **攻城**：`- Defeat: Hall lost if your Army rents one`。结盟/宣战属于这一层。

### P4 · 经济与玩家互动

**银行 → 邮件（带附件）→ 中古商店（寄卖）** 三步。注意邮件顺带解决我们在 Decision Log 里搁置的
「不允许离线补领奖励」——邮件就是这个问题的标准答案。中古商店 = 我们暂缓的拍卖行的轻量形态。

### P5 · 运营层

限时 buff / booster（红月 21+9 个）、事件调度器（双倍经验、生存竞技场）、排行榜、硬核模式。
排行榜特别值得注意红月的选择：**天梯排的是奖金点数，不是等级**——因为等级会被转生清零，
需要一个跨转生的累计量。我们的转生系统有同样的问题，同样需要这个量。

## 6. 明确不抄的东西（反面教材）

参照仓库里有一批**必须反向学习**的东西，我们现在的做法已经更好，不要在"对齐参照"时倒退：

- **明文密码**：`tblBillID.Password` 直接明文比对。我们用 SHA-256 摘要 + 恢复码 + 轮换。
- **商店零校验**：商店只靠手打 Game ID 认人（无登录），且客户端 POST 一个 `paypal_success`
  服务端就发货，**没有任何服务端侧 PayPal 验证**。这正是我们「客户端只提交意图」红线要防的东西。
- **buff 存成手撕字符串**：`SETimer = '3:24853705.7:24557789.…'` 塞进 varchar。P5 做限时 buff 时用结构化存储。
- **字段名说谎**：转生计数寄生在 `SubQuestGiftFame` 上，订阅到期寄生在 `PoisonUsedDate` 上。
- **排行 N+1**：3000 行循环里逐行 `SELECT`，无缓存；GM 账号靠**硬编码名字黑名单**过滤。
- **转生并发靠临时文件锁** + 「重试 3 次遇到 deadlock」，而不是数据库事务。

## 7. 待人拍板（Open questions）

1. **P0 转生门槛定多少？** 直接对齐红月的 1000 级？还是保留低门槛但改成递减收益？这是数值口味问题，需要人定。
2. **#1 PostgreSQL 闸门什么时候算过？** P2 之后全部依赖它。
3. **一号多角要不要做？** 现在「名字即账号 + 永久绑定一个职业」意味着玩家想试第二个职业就得换账号，
   8 个职业的设计投入被这条规则浪费了大半。改动是 schema 级的，越晚做越贵。
4. **PvP 的隔离程度？** 红月选择「独立战斗地图」而非全图开放。要不要照抄这个保守选择？
