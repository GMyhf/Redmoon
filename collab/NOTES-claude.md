## T-017 P2 第二步 · 荣誉（Claude → Codex）· 待复核 · **原计划被我推翻，请审这个判断本身**

**最该看的是这个决定，不是代码。** 计划原文写「把 Eclipse 专属的 `reputation` 泛化为全局属性、PvP 击杀驱动」。
我读完代码后认为**这个设想有两处硬伤，不能照做**：

1. **荣誉在第二步没有来源** → 会交付一个死数字。决斗**明确不给荣誉**（而且必须不给：双方同意的 PvP
   可以两个朋友互喂刷分），野外 PvP 是第三步。照计划做，第二步会上线一个 ±1000、持久化、显示在 HUD 上、
   **永远不动**的数字——**正是我在 P0 指着 `will` 说「这是死资源」的那个形状**。我不想自己再造一个。
2. **`reputation` 不是「Eclipse 版的荣誉」**。它是玩家**主动选择**的构筑轴：`attune` 选光辉/深渊，
   每次施法推 ±2，**符号翻转会切换整套技能**。让 PvP 战绩驱动它 = 玩家的技能组会和他自己选的归属打架。
   那不是泛化，是两个无关语义撞在同一个字段上。

**人选了方案 A**（独立字段 + PvE 来源 + 门禁消费方）。依据不是我拍脑袋，是**红月自己的字符串表**：
```
Killing Red monsters generates positive karma!        ← 荣誉有 PvE 来源
Honor is insufficient to upgrade.                     ← 门禁精炼
To create army you must be at level %u and honor %u   ← 门禁军团（P3 前置）
You cannot use this skill with your current Honor Points.
  - HONOR        : Reputation                          ← 红月自己把两者释义为同一个词
```
**红月的荣誉不是 PvP 记分板，是一套权限系统。** 最后那行说明我原计划想合并两者的直觉在命名上有出处——
但我们的 `reputation` 已经被 Eclipse 占用为另一套机制了，所以不能合。

**核心设计点**：**荣誉是门槛，不是货币——只检查，从不扣除。** 这样它不制造第二条 grind，只要求「你真的去猎过」。
我为这一点专门写了断言（`assert.equal(player.honor, honorBefore, "标准不是价格")`），并做了变异测试：
把它改成会扣除 → 用例立刻挂。

**变异测试 ×3 全部命中**，其中第三条专门守我推翻原计划的那个理由：
让击杀同时驱动 `reputation`（= 原计划的做法）→ `honour and Eclipse's reputation are different numbers` 立刻挂。

**我的取舍，想被质疑的**：
1. **荣誉暂不公开给其他玩家**（只在自身快照里）。红月的 10 档配色是给别人看的，但那要等野外 PvP 让它
   有社会意义（谁是杀人犯）才成立；而且公开就要动 `PLAYER_BASE` + binary codec → 升协议。
   **我选择把这个协议改动推迟到第三步，和战场字段一起升。** 你可能认为现在就该公开。
2. **门槛值 200/400 是我定的**，人只拍了方案 A。前两阶不设门槛（新装备永远能改），只卡 +3/+4。
   200 荣誉 = 200 只精英或 40 个 Boss。**可能太松（等于没卡），也可能太紧。**
3. **精英 +1 / Boss +5 的比例是我定的。**
4. `HONOR_TIERS` 我只定义了正向档位（无名/闻名/信重/威名/传颂）。负向档位等第三步有了负向来源再加——
   现在定义就是死配置。

**验证**：`npm test` **189/189**（新增 5 条，独立文件 `test/server-honor.test.js`）｜ `npm run check`
｜ `npm run check:godot` ｜ **变异测试 ×3** ｜ **真协议端到端 4/4**（精英击杀发 `honorChanged`、
**快照里 honor=1**（上次精炼栽过的字段挑选器坑）、荣誉不足被 `NOT_ENOUGH_HONOR` 拒、荣誉够了通过且未被扣除）。

## T-016 P2 第一步 · 决斗场（Claude → Codex）· 待复核 · **PvP 首次落地**

**这是全仓第一次让玩家能伤害到玩家**（此前 `pvp` 零命中，玩家投射物只检查怪物）。
人已拍板 #1 闸门通过、P2 排期（Decision Log 2026-07-17）。按计划分三步，这是第一步：
**双方同意、隔离地图、无掉落、无经验、无荣誉——只验证「伤害能路由到玩家」这条链路。**

**请你重点当红队打的地方——隔离。** 我的实现是：投射物只在决斗地图上、且**只对该决斗 `members` 列表里的
那一个对手**做碰撞。不是按距离扫玩家，也不是「同图的都能打」。三条泄漏我都写了测试，但**你能想到第四条吗**：
1. 普通地图上两个玩家互射 → 打不到；
2. 决斗中的人朝另一张图上站在同坐标的旁观者射 → 打不到；
3. 场外第三人不会被选为对手（`members` 只有两个 id）。

**两个不在原计划里、实做才发现的硬点**：
1. **竞技场必须有自己的边界**。四条移动路径（键盘移动、点击移动、技能位移、moveTo 目标）**全都**钳制到
   `this.width/this.height`（4800×2700）。不改的话决斗者能一路跑到空旷的世界另一头——**竞技场没有墙**。
   已加 `_boundsFor(entity)` 收口这四处。
2. **对手解析方式决定了隔离是否成立**。见上。

**变异测试的一段经历，我觉得比结果本身值得说**：
我给「跨地图打不到」写的第一版测试，**连着三次变异都抓不住**——① 把 PvP 改成全图开放：过；
② 把对手解析改成「取第一个非自己的玩家」：过；③ 改成「遍历所有玩家碰撞、忘记过滤 mapId」（经典写法）：**还是过**。
原因：我把旁观者和对手放在同一坐标，而普攻只有 1 次命中，**弹被对手先吃掉了，根本到不了旁观者**。
那条测试从头到尾是摆设。重写成「对手挪出弹道之外，弹道上只有另一张图的旁观者」后，第 ③ 种变异立刻挂。
**教训：断言「A 没被打到」时，必须确认 A 真的处在会被打到的位置上。** 现在三条安全断言全部经过变异验证。

**我没把握 / 想被质疑的**：
1. **`detachPlayer` 即判负**。断线保席机制给了 5 分钟宽限（队伍/副本席位都保），但决斗我选择立刻判负——
   否则一个人拔网线就能把竞技场冻住 180 秒。**代价是网络抖动会直接输掉一场。** 这个取舍值得你挑。
2. **决斗中允许喝药**（药在背包里，我没禁）。这让决斗变成「谁药多」而不是「谁强」。要不要禁？
3. **`_endDuel` 把双方送回 `origins` 记录的进场前位置**，包括进场前所在的地图。如果那张图在决斗期间
   出了什么事（理论上不会，地图是静态的），玩家会被放到一个奇怪的地方。我认为可接受，但这是个假设。
4. **没做等级差限制**——1000 级打 1 级也可以，因为双方都同意了。你可能认为该拦。

**验证**：`npm test` **184/184**（新增 13 条，独立文件 `test/server-duel.test.js`）｜ `npm run check` 通过
｜ **变异测试 ×4**（含上面那段失败经历）｜ **两个真浏览器端到端**：邀请 → 应战 → 进场 → 认输 → 回城，

## T-011 P1 装备精炼 + 护炉印（Claude → Codex）· 待复核 · **协议 v3**

**背景**：`docs/P1_REFINE_SPEC.md` 是规格，`docs/IMPROVEMENT_PLAN.md` 是全局计划。数值口味由人拍板：
曲线 90/70/50/30、失败掉阶、护炉印收复苏露。

**这轮最值得你看的一件事**：我在端到端验证里抓到一个**测试和 conformance 都抓不到的真 bug**。
`world.js` 的 `serializeItem` 和 `publicItem` 是**显式字段挑选器**，新加的 `item.refine` 不在名单里，
于是精炼阶数**根本传不到任何客户端**——服务端 item 改了，快照里没有。为什么没被抓到：
- 我的单元测试断言的是 `item.refine`（服务端对象），不是快照；
- protocol conformance 里 `refine: "number?"` 允许字段缺失，**缺了也算合规**。
