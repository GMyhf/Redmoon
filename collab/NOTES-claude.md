## T-012 R/C 职业专属化（Claude → Codex）· 待复核 · **协议未变**

**任务原本写的是「16 个换皮技能，改成专属是纯粹的品质提升」。实做时发现它其实是个数值 bug**，
这是本轮最重要的一条：

`shared:r` 按 `力量+精神` 成长、`shared:c` 按 `敏捷` 成长，但自动加点按 `ALLOC_WEIGHTS` 分。算出来：

| 职业 | 加点进敏捷的比例（= 旧 `C` 的属性成长） |
| --- | --- |
| 壁垒者 | **8.8%** |
| 焚灵 | **9.6%** |
| 先锋 | 16.1% |
| 导能者 / 蚀者 | 17.5% |

**五个职业的 `C` 属性加成近乎失效**——不是手感问题，是这个技能根本不随角色成长。
现在 16 个槽位各自拥有行为、按本职业主属性成长，全职业最低占比 8.8% → **35.1%**（壁垒者，因为它加点偏体魄）。

**改动很小**：`_castBehavior` 本来就是 `SKILL_BEHAVIORS[\`${archetype}:${slot}\`] ?? SKILL_BEHAVIORS[\`shared:${slot}\`]`，
所以**加键即覆盖，`world.js` 一行没动**。`shared:r`/`shared:c` 保留为兜底（新职业忘了配时不会崩），
并且我把它们的注释改成了「fallback only」。

**保持不变的**：R/C 的定位（R=重击、C=位移+环）、冷却（8.5s / 10s）。本次只动形态与属性成长，
不同时移动平衡面——冷却要不要按职业分化，是下一步的事。

**请重点看 / 想质疑的**：
1. **16 组数值是我按各职业的 q/e 倍率「同族」推的**，没有做战斗模拟验证。比如先锋 q 是 power×2.1、
   e 是 ×1.25，我给 r ×1.6、c ×1.5。这是手感推断，不是数学论证。**值得你挑**。
2. **壁垒者的 `C`「守望铁环」我做成了全场唯一不位移的 C**（用最大最密的环换掉逃生）。
   这是有意的角色区分——坦克守阵地——但它**删掉了一个职业的唯一位移手段**，是本轮最激进的一处。
   端到端量出来：壁垒者 C 位移 0px，其他职业 108-280px。**如果你觉得这剥夺了必要的生存工具，说服我。**
3. **`shared:r`/`shared:c` 现在是死代码**（8 个职业全覆盖了）。我选择保留作兜底而不是删除，
   理由是新增职业忘配 r/c 时会静默拿到 `undefined` 行为 → `_castBehavior` 崩。但也可以论证应该删掉、
   让缺配置在测试里响亮地失败（我加的 `every archetype owns its R and C` 用例正是干这个的）。**你怎么看？**

**顺带发现、我没动的（既有问题，不在本任务范围）**：
**技能栏中英混排**。先锋实际显示 `Q: Ram Drive / E: Resonant Ring / R: 裂阵重斩 / C: 赤钢回旋 / F: Skybreaker`——
q/e/f 是英文、r/c 是中文（中文名在 `EXTRA_SKILL_NAMES` 里本来就有，我沿用了）。
而且 `docs/GAME_TUTORIAL.md:75` 写「先锋『绯岚断空』」，但代码里叫 `Skybreaker`，**教程描述了一个不存在的本地化 UI**。
物品有客户端 `ITEM_NAMES` 翻译表，技能没有。正解应该是给技能补一张同样的表（涉及全部 40 个技能），
属于本地化范畴，**我没有在这条任务里悄悄扩**。建议单开一条。

**验证**：`npm test` **169/169**（新增 3 条）｜ `npm run check` 通过 ｜ **变异测试**：把 16 个专属键改名让查找回落到
`shared:*`，三条新用例全部 `not ok`，恢复后全过（第一次我用正则删键，正则没匹配上、16 个键原封不动，
那轮「通过」是无效的——重做后才是真的）｜ **端到端**：八职业逐个施放 R/C，弹数 1-5 各异、C 位移 0-280px 各异。

## T-011 P1 装备精炼 + 护炉印（Claude → Codex）· 待复核 · **协议 v3**

**背景**：`docs/P1_REFINE_SPEC.md` 是规格，`docs/IMPROVEMENT_PLAN.md` 是全局计划。数值口味由人拍板：
曲线 90/70/50/30、失败掉阶、护炉印收复苏露。

**这轮最值得你看的一件事**：我在端到端验证里抓到一个**测试和 conformance 都抓不到的真 bug**。
`world.js` 的 `serializeItem` 和 `publicItem` 是**显式字段挑选器**，新加的 `item.refine` 不在名单里，
于是精炼阶数**根本传不到任何客户端**——服务端 item 改了，快照里没有。为什么没被抓到：
- 我的单元测试断言的是 `item.refine`（服务端对象），不是快照；
- protocol conformance 里 `refine: "number?"` 允许字段缺失，**缺了也算合规**。

只有真起服务器 + ws 客户端读快照才暴露。已补 `the refine stage reaches clients through the snapshot`
用例（断言快照，不是内存对象），并做了变异测试：撤掉序列化修复 → 该用例 `not ok`。
**建议你重点想想：还有哪些「新字段进不了挑选器」的同类风险？** 这个模式在这仓库里会重复咬人。

**改动要点**：
- 精炼倍率只作用在 `_refreshGear` 这**一个汇总点**上（`scale = 1 + 0.15*stage`）。
  **存下来的 bonuses 永不改写**——否则精炼会变成刷词条。
- `itemPower`（服务端）与 `itemScore`（客户端）都乘了精炼倍率，否则自动换装/↑↓ 箭头会让玩家把 +4 装备扔掉。
- 护炉印做成 `player.protections` **计数器**而非背包物品：背包**不支持堆叠**，囤 8 张要占 8 格。
  为此 `buyGood` 加了显式分支（原来的 `else` 会直接掉进 `_rollRelic`）。
- **可精炼身上的装备**：`equipItem` 会把物品 splice 出背包，所以 `refineItem` 同时查背包和 equipment，
  否则要玩家做「脱→炼→穿」的无谓动作。
- 两个客户端的**面板签名 bug**：浏览器 `updateGear` 的 signature 和 Godot 的 `bag_signature` 原本只含 item id，
  精炼只改 `refine` 不改 id → 面板永不重绘。两处都把阶数并进签名了。
  浏览器那处我**故意没把 `will` 并进签名**——它每次击杀都变，会导致整个背包每杀一只怪就 `replaceChildren` 一次。

**我没把握 / 想被质疑的**：
1. **`REFINE_STEP = 0.15` 满阶 +60% 的强度**，以及 `REFINE_WILL_PER_LEVEL=4` / `REFINE_GOLD_PER_LEVEL=6`
   的费率——这三个数字**我是拍脑袋定的**，规格里也标了未决。人只拍板了曲线/掉阶/货币三项。
   消耗池够不够深、+60% 是不是分水岭，值得你算一算。
2. **`purchased` 事件的 schema 我改松了**：`itemId`/`name`/`rarity` 从必填改成可选（`"string?"`），
   因为护炉印不产生物品。代价是真物品购买漏发这些字段也能过 conformance。
   另一个选择是给护炉印单开一个事件类型。**你觉得哪个更好？**
3. **精炼门槛 `tier >= 3`**：sunset 是 tier 7 / 0.9% 掉率 / 全服地上同时只准 1 个，只让它可精炼等于锁死成
   没人碰得到的终局内容。所以放到 rare+。但这也意味着前期 rare 装备就能进消耗池，可能过早。
4. **失败掉阶但护炉印在 +0 时也会被消耗**（我在代码注释里写明了）：+0 本来无阶可掉，带印去炼 +0→+1
   等于白烧一张。要不要在服务端直接拒绝 +0 带印、或静默不消耗？我选了「照常消耗」因为它最不意外，
   但这对玩家不友好。

**验证**：`npm test` **166/166**（新增 7 条）｜ `npm run check` 通过 ｜ `npm run check:godot` 通过（exit 0）
｜ **变异测试** ×2（转生闸门、序列化修复）｜ **真协议端到端** 5/5：`welcome.protocol=3`、
护炉印买入不占背包格、rng 0.5 精炼成功进 +1、**快照里 refine=1**、普通装备被 `REFINE_TIER_TOO_LOW` 拒。

**顺手修了一个既有 flaky 测试（不是本任务引入的，但污染绿灯闸门，同 T-002 性质）**：
`test/browser/ui.test.mjs` 的副本用例 `dungeon entry starts the child worker`。
我先怀疑是自己的回归（第一轮 A/B：我的分支 0/3、干净 main 2/3），加探针后定位到真因并**证伪了这个怀疑**：
- 探针数据：`mapId` 是**同步**翻的，离开按钮只跟 `mapId` 走，而 worker 子进程 spawn + 握手要 **~240ms**；
  按钮到断言之间只有 250-550ms → 抛硬币。
- **带同一探针时，干净 main 2/2 全挂、我的分支 1/2 通过** —— 第一轮的 0/3 vs 2/3 是本机负载噪声。
  这台机器无 GPU、跑着 Chrome，正好落在硬币的边上。
- 修法：`await waitForServer(() => Boolean(dungeon.workerTransport))`，和这套件里其他异步等待一致。
  **断言本身没动、没放宽**，只是不再对一个异步过程做零等待断言。修后连跑 **3/3**。

浏览器全套本机每例约 40 秒（19 例 >10 分钟），结果见 HANDOFF。

## T-010 P0 转生门槛与 will 定位（Claude → Codex）· 待复核

**背景**：新增 `docs/IMPROVEMENT_PLAN.md`——对照 `/home/ubuntu/hongfei/RedMoon-Fantasy`（红月 3.944e 私服发行仓库）
盘的差距与分阶段计划。这条是其中 P0 的前两项，数值口味由人拍板（门槛=1000、收益不变无限叠加；will 留作 P1 出口）。

**改了什么**：`REBIRTH_LEVEL` 10 → `LEVEL_CAP`（1000）。收益公式一个字没动。

**为什么这是 bug 不是调平衡**：10 级只要 3,544 经验（约 32 只怪），而每次转生的
`+6 属性点 / +12% 生命 / +15% 伤害` 是**永久且无上限叠加**的。原来的组合允许在灰港边上刷 32 只怪 → 转生 → 再刷 32 只
→ 无限循环，几小时就能把数值叠到任何设计意图之外，1→1000 的等级阶梯（3.33 亿经验）等于白设计。红月把门槛放在
1000 正是因为奖励永久——两者是一套的，我们只抄了奖励没抄门槛。

**请重点看这几处（我没把握 / 想被质疑的）**：

1. **`REBIRTH_LEVEL = LEVEL_CAP` 这个绑定合不合适**。我把 `LEVEL_CAP` 的定义位置上移到了转生常量组上方才能这么写。
   语义上「转生 = 满级后的唯一出路」我认为成立（满级后 `_grantXp` 直接 early-return，玩家确实无事可做），
   但如果将来想让门槛低于上限，这个绑定就得拆开。**如果你觉得该保持两个独立常量，说服我。**
2. **`_grantXp(player, 400_000_000)` 这种测试写法**。为了把玩家推到满级，测试里一次灌 4 亿经验，`_grantXp`
   会循环 1000 次（每次 `_refreshDerivedStats`）。跑起来没有可观耗时，但这是不是比直接 `player.level = LEVEL_CAP`
   更好？我选灌经验是因为它走的是真实升级路径，能顺带证明升级循环能一路跑到 cap；但它也让测试依赖 XP 曲线的具体数值。
3. **门槛提高后转生在原型期可能是死内容**。人已经知情并拍板（选项里明确写了「3.33 亿经验，原型期大概率无人触及」），
   但你如果有更好的兼顾办法（例如可配置门槛用于测试服），值得提。

**验证**：除 `npm test` 159/159 外，我做了两件事：
- **变异测试**：把常量改回 10，新测试 `not ok`（`expected: true, actual: false`）——确认它是真闸门，不是摆设。
- **真协议端到端**：起真实 `GameServer` + ws 客户端跑通：`welcome.rebirthLevel = 1000` ✅；
  10 级发 `rebirth` → `REBIRTH_LEVEL_TOO_LOW` 且 `rebirths` 仍为 0 ✅；1000 级发 `rebirth` → `rebirths=1, level=1`
  且无 error ✅。

**两处我故意没动，需要你/人定**：

- **`docs/CRIMSON_RELAY_NOVEL.md:96`**：「他们各自摸到了**第十级**。中继站的水晶屏上第一次跳出一个陌生的选项：**转生**。」
  这句现在与玩法矛盾。我没改，两个理由：① 这个文件是你 T-009 正在 Review 的文件，硬约束不允许我同时动；
  ② 单纯把「第十级」换成「满级」会牵动时序——转生现在是终局，却发生在第四章（第三章锈王之后、第五~八章之前）。
  这是叙事取舍，不该我一个人拍。**建议 T-009 落地后单开一条任务处理。**
- **`will` 在 P0 只补了注释**（`world.js` 的字段注释 + CHANGELOG + 计划文档），代码零改动。
  它仍然是「全职业累加、仅 Eclipse 可见（`client.js:830` 的 alignmentRow 对其他职业隐藏）、无任何作用」。
  人选择留作 P1 精炼货币而非删除——删它要动 `codec.js` 的 u32 并升 `PROTOCOL_VERSION` 到 3，P1 若又需要击杀货币
